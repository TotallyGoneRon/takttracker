    **Query params:** `planId` (required, integer), `walkId` (required, integer -- current walk to exclude from trend)

    **Response shape:**
    ```typescript
    {
      previousWalk: {
        id: number;
        walkDate: string;
        onTrackRate: number; // percentage 0-100
      } | null;
      nextUpTasks: Array<{
        taskName: string;
        zoneName: string | null;
        companyName: string | null;
        plannedStart: string;
      }>;
    }
    ```

    **Previous walk logic (SUM-04, per D-09 and D-10):**
    1. Query `siteWalks` where `takt_plan_id = planId`, `status = 'completed'`, `id != walkId`
    2. Order by `walk_date DESC`, limit 1 to get most recent completed walk
    3. If found, query its entries from `siteWalkEntries` where `site_walk_id = previousWalk.id`
    4. Calculate onTrackRate: `Math.round(((on_track + completed) / total) * 100)`
    5. If no previous walk found, return `previousWalk: null`

    **Next-up tasks logic (SUM-03, per D-07 and D-08):**
    1. Import `getToday` from `@/lib/dates` for today's date
    2. Calculate endDate as today + 3 calendar days (use `new Date()` arithmetic, format as YYYY-MM-DD)
    3. Query tasks table:
       ```typescript
       db.select({
         taskName: tasks.task_name,
         plannedStart: tasks.planned_start,
         zoneName: zones.name,
         companyName: companies.name,
       })
       .from(tasks)
       .innerJoin(activities, eq(tasks.activity_id, activities.id))
       .leftJoin(zones, eq(tasks.zone_id, zones.id))
       .leftJoin(companies, eq(activities.company_id, companies.id))
       .where(and(
         eq(tasks.takt_plan_id, planId),
         gte(tasks.planned_start, today),
         lte(tasks.planned_start, endDate),
         eq(tasks.is_trackable, true),
       ))
       ```
    4. Filter out completed tasks in memory: `.filter(t => t.taskStatus !== 'completed')` -- include tasks.status in the select for this filter, then omit it from the response
    5. Per research Pitfall 3: Group by company+taskName to avoid massive lists. Return one entry per unique (companyName, taskName) combination with the earliest plannedStart from that group, plus a count field if >1 task. Actually, keep it simple per D-07 which says "each entry shows task name, zone, and planned start date" -- return individual tasks but cap at 50 results max.
